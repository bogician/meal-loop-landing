## Deferred from: code review of 2-3-og-social-share-image (2026-06-22)

- next-intl proxy injects `Set-Cookie: NEXT_LOCALE` + hreflang `link` headers onto the OG image response (`src/proxy.ts:15`). The emitted `og:image` URL has no file extension (`/{locale}/opengraph-image?<hash>`), so the dot-exclusion matcher `/((?!api|_next|_vercel|.*\..*).*)` does not skip it — every crawler fetch of the social-preview image runs through locale negotiation and gets a pointless `Set-Cookie`. Harmless (crawlers ignore it) but wasteful. Fix = tighten the proxy matcher to also skip `opengraph-image`; out of Story 2.3 scope (touches the shared proxy, affects all routes).
- No `try/catch` around `readFile` in `src/app/[locale]/opengraph-image.tsx:36`. A future locale added to `LOCALES` without a committed `public/og/og-<locale>.png` would throw `ENOENT` during the `generateStaticParams` build prerender and fail the build, rather than degrading. Arguably correct fail-fast-at-build behavior; both current PNGs (en/uk) are present, so not actionable now. Revisit only if locale-asset coupling becomes error-prone.
- `x-default` hreflang alternate (auto-emitted by next-intl) resolves to the bare `/opengraph-image`, which `307`-redirects to `/en/opengraph-image` before serving the PNG. Stricter OG scrapers that do not follow redirects on image URLs would miss the x-default variant; the per-locale `og:image` tags themselves are direct (no redirect), so impact is low. Belongs with Epic 3 hreflang/metadata work, not Story 2.3.
